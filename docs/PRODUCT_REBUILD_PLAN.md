# CryptoOrder 重构方案（0→1：交易 + 笔记 SaaS）

## 1. 产品定位

### 1.1 目标用户（ICP）
- 高频/中频合约交易者（以 BTC/ETH 永续为主）。
- 有交易纪律意识，但复盘体系不完整。
- 愿意为“稳定提升胜率与执行力”付费。

### 1.2 核心问题
当前市面交易工具聚焦“下单效率”，缺少“成长效率”：
1. 交易后无法结构化复盘。
2. 无法识别长期错误模式（追单、报复性交易、过度杠杆）。
3. 无法量化“计划 vs 执行”偏差。

### 1.3 产品价值主张
> 不是只让用户交易更快，而是让用户进步更快。

通过 **交易执行 + 自动日志 + 复盘洞察 + 改进建议** 的闭环，帮助用户持续优化策略与行为。

---

## 2. 产品目标与指标

### 2.1 北极星指标（NSM）
- 每活跃用户每周完成的高质量复盘条数（HQ Reviews / WAU）。

### 2.2 核心业务指标
- 激活：注册后 24 小时内完成首笔“交易+笔记”闭环比例。
- 留存：D7 有至少 1 条交易笔记用户比例。
- 转化：累计 20 条交易笔记用户的 Pro 付费率。

### 2.3 质量指标
- 交易事件入库成功率 ≥ 99.9%。
- 日志自动生成时延 ≤ 3 秒。
- 统计计算任务成功率 ≥ 99%。

---

## 3. 需求分层（Scope）

### L1 执行层（Trade）
- 实时行情、下单、持仓、平仓、风险展示。

### L2 记录层（Log）
- 自动生成交易日志（含开平、手续费、滑点、持仓时长、MAE/MFE）。
- 手动补充：入场理由、策略标签、情绪、计划止损、执行评分、复盘结论。

### L3 洞察层（Insights）
- 胜率、盈亏比、期望值、最大回撤、时段表现、标签表现。
- 计划/执行偏差统计。

### L4 改进层（Coach）
- 周报与错误模式识别。
- 行为建议（如“连续亏损后杠杆上升风险”提示）。

---

## 4. MVP 定义（12 周内）

### 4.1 必须做（Must）
1. 交易工作台（模拟盘优先，可扩展实盘）。
2. 自动交易日志生成。
3. 交易详情 + 笔记编辑。
4. 日志列表与筛选（时间、方向、标签）。
5. 基础复盘面板（胜率、盈亏比、净收益、最大回撤）。

### 4.2 应该做（Should）
1. 周报自动生成。
2. 策略标签维度分析。
3. 图表截图与复盘附件。

### 4.3 可选（Could）
1. AI 复盘总结。
2. 团队协作复盘（Team 版）。

---

## 5. 用户旅程（Core Journey）

1. 用户注册登录。
2. 连接交易账户或进入模拟模式。
3. 在交易工作台执行交易。
4. 系统自动创建交易日志草稿。
5. 用户补充主观字段（理由/情绪/复盘结论）。
6. 用户在 Insights 查看统计与行为模式。
7. 每周收到改进建议，形成下一周计划。

## 7. 功能需求（精简 PRD）

## 7.1 模块 A：交易工作台

**用户故事**
- 作为交易者，我希望在一个页面内完成看盘、下单、查看持仓，避免切换损失时机。

**需求点**
- 支持标的切换、杠杆设置、仓位输入、开多/开空、平仓。
- 展示可用余额、占用保证金、未实现盈亏。
- 记录每次操作事件（用于审计与复盘）。

**验收标准**
- 下单成功后 3 秒内可在日志中心看见对应记录。
- 仓位与盈亏展示与交易事件一致。

## 7.2 模块 B：交易日志中心

**用户故事**
- 作为交易者，我希望每笔交易自动带出关键字段，并可补充主观信息，减少复盘成本。

**字段定义**
- 自动字段：symbol、side、entry/exit、qty、fee、slippage、duration、mae、mfe、realized_pnl。
- 手动字段：setup_tag、entry_reason、emotion、plan_sl、actual_sl、execution_score、lessons_learned。

**验收标准**
- 日志列表支持按时间、标签、方向过滤。
- 日志详情编辑保存成功率 ≥ 99.9%。

## 7.3 模块 C：洞察仪表盘

**用户故事**
- 作为交易者，我希望知道自己在哪些策略和时段表现最好，从而优化仓位与频率。

**指标**
- 总交易数、胜率、盈亏比、净收益、最大回撤、期望值。
- 按标签/时段/方向聚合表现。

**验收标准**
- 指标计算每日自动更新。
- 支持最近 7/30/90 天切换。

## 7.4 模块 D：周报与改进建议

**用户故事**
- 作为交易者，我希望系统告诉我“下周最该改什么”，而不是只看静态图表。

**需求点**
- 生成周度总结：亮点、问题、行动建议。
- 输出最多 3 条可执行建议（可打勾跟踪）。

**验收标准**
- 每周一自动生成上周报告。
- 建议可标记“已执行/忽略”。

## 8. UI 设计原则

1. 信息密度高但层次清晰（交易区 / 记录区 / 复盘区）。
2. 语义色统一：盈利绿、亏损红、提醒黄。
3. 关键操作低摩擦（交易后 2 步内完成笔记）。
4. 桌面端优先，移动端保障查看与轻编辑。

---

## 9. 技术架构（程序设计）

## 9.1 前后端建议
- Frontend：React + TypeScript + Zustand + TanStack Query。
- Backend：NestJS（或 FastAPI）。
- DB：PostgreSQL。
- Cache/Queue：Redis + BullMQ。
- Realtime：WebSocket。

## 9.2 服务拆分
1. 行情服务（Market Data Service）。
2. 交易事件服务（Trade Event Service）。
3. 日志服务（Journal Service）。
4. 指标计算服务（Analytics Service）。
5. 报告服务（Report Service）。

## 9.3 非功能需求
- 安全：API Key 加密、最小权限、审计日志。
- 稳定：核心路径重试与幂等。
- 观测：日志/指标/链路追踪。

## 12. 风险与应对

1. **交易数据一致性风险**：通过事件溯源 + 幂等键保障。
2. **用户不写笔记风险**：提供自动草稿与模板化输入。
3. **分析价值不足风险**：优先做“可行动建议”而非纯图表。
4. **付费转化低风险**：在“20笔后诊断报告”节点触发升级。
